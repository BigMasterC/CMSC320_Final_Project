Introduction

This tutorial focuses on Airbnbs, and determining correlation among features. As Airbnbs become more popular, it becomes more important to know what certain data about an Airbnb can mean in regard to other features, especially when those other features (e.g. price) are not available at that moment and could sway one's decision to stay at an Airbnb. One question this tutorial seeks to answer is if an Airbnb's rating can be used to predict its price. If this is possible, it will help Airbnb customers ensure they are paying fair prices by checking the Airbnb's rating. Another question this tutorial seeks to answer is if an Airbnb's check-in time can predict its check-out time. If this is possible, it will help customers determine how long they will be allowed to stay at a certain Airbnb (even if that information is not made available), so they can ensure they are paying fair prices for the amount of time they are allowed to stay.

Data Preprocessing

Many of the rows contain a non-numerical rating of "New." Possible options for fixing this issue include dropping the rows with a rating of "New" or using the mean/median for rows missing a rating. However, since ~33% of the data has a "New" rating, those options were less desirable. Therefore, the best option was to use a regression model with existing data to predict the ratings for rows missing rating values, and use those predictions to fill in the missing data.

data_processing.py 

# Initializing variables
X = df[df['rating'] != 'New']
y = X[['rating']]
X = X[['price', 'bathrooms', 'beds', 'guests', 'toiles', 'bedrooms', 'studios']]
seed = 42
model = SVR(kernel='rbf')

# Splitting the data
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2,
                                                    random_state=seed)
# Training the model
model.fit(X_train, y_train)

# Using the model to make predictions for rows missing data
for row in df.itertuples(index=True):
  if row.rating == 'New':
    row_data = {
        'price' : [row.price],
        'bathrooms' : [row.bathrooms],
        'beds' : [row.beds],
        'guests' : [row.guests],
        'toiles' : [row.toiles],
        'bedrooms' : [row.bedrooms],
        'studios' : [row.studios],
    }
    prediction = model.predict(pd.DataFrame(row_data))
    df.at[row.Index, 'rating'] = round(prediction.item(0), 2)

Cleaning up the Check in/Check out time

Adds two new intermediary columns and two new final columns checkin_time_clean and checkout_time_clean

Airbnb "check-in" and "check-out" times are user-entered, messy, and inconsistent. Examples include:


  • "12:00 pm"
  • "4pm - 11pm"
  • "After 1 pm"
  • "flexible"
  • "Depends"
  • missing entries (NaN)
  • country differences in typical times

These cannot be directly analyzed unless they are converted to uniform, numeric values.

In this preprocessing, we seek to normalize all the times into a datetime format. To do this, we have to normalize all these odd outputs (e.g., "4 00 pm", "after 2pm", "4pmâ€“11pm"), handle "flexible" checkins (rather than just discarding them), and handle missing data without bias.


A few assumptions were made so that this would be doable:


  1. Checkin ranges indicate the earliest allowable checkin.
    • Airbnb hosts typically list a range meaning "guests can check in anytime between these bounds," but for modeling and comparison, we need a single time point.
    • We assume the earliest time in the range is used as the functional check-in time.
  2. "Flexible" check-in reflects typical check-in behavior of the country
    • The term "flexible" is ambiguous and varies across hosts. It is not interpretable as a specific time. So, we replace "flexible" with median checkin times of that country.
    • We assume that host flexibility roughly aligns with norms in their region, local market structure and guest expectations influence typical times, and country-level medians approximate expected check-in availability. This avoids imposing a universal assumption like 3 PM, which may not generalize globally.
  3. Missing values are not missing at random (MNAR)
    • Many missing check-in/out times occur because hosts leave defaults, skip the field, or specify details elsewhere in the listing.
    • We assume that missing values reflect considerations similar to flexible entries and should be replaced using country-level medians.
data_processing.py 
def parse_time_text(text):
    """Parse a text like '4 00 pm' or '11 am' into a datetime.time.
    Handles formats like:
        - '4 00 pm'
        - '11 am'
        - '3pm'
        - '12 30 am'
    Returns None if parsing fails.
    """

    # Parses to see if the particular text matches with a time format
    match = re.search(r'(\d{1,2})\s*(\d{2})?\s*(am|pm)', text)
    if not match:
        return None

    # Obtains the hour, minute, and meridian of the text
    hour = int(match.group(1))
    minute = int(match.group(2)) if match.group(2) else 0
    meridian = match.group(3)

    # Converts to 24 hour time
    if meridian == "pm" and hour != 12:
        hour += 12
    if meridian == "am" and hour == 12:
        hour = 0

    # Returns the time as a datetime object
    return dt.time(hour, minute)

2. Is there a linear correlation between price and rating?

To execute a hypothesis test, we must first define the null and alternative hypotheses:

  • H0: There is no linear correlation between price and rating.
  • HA: There is a linear correlation between price and rating.

We also define our significance level to be α = 0.05. If we determine that p < α, then we reject the null hypothesis.


Since we are trying to determine if price and rating are linearly correlated, the best test to use is the pearson correlation coefficient.

The p-value is approximately 2.332*10-12, which means there is a 1/(2.332*1012) chance of us this pearson correlation between price and rating if we fail to reject the null hypothesis. 3. Does the Rating of an Airbnb Have an Effect on the Number of Guests the Airbnb Can Accommodate?

We just discovered that there is a relationship between the price per night of an airbnb and its rating. Now, let's explore the question of whether or not the rating of an airbnb affects the number of guests accommodatable by that airbnb. Do airbnbs with higher ratings typically have room for more guests than airbnbs with lower ratings, or is it the other way around? If so, does this mean the price per night indirectly affects the rating? Or, does rating even matter when considering the number of guests that can live in an airbnb? Let's find out!


Let's start by defining the null and alternate hypotheses:

  • H0: The rating does not have an effect on the number of guests.
  • Ha: The rating does have an effect on the number of guests.

First, we need to determine the general characteristics of the values in the rating column. How spread out are they? Are the ratings skewed? Let's do some EDA on the ratings.

From the exploratory data analysis above, we can see that the distribution of ratings is heavily left skewed, with the majority of ratings being from 4.5 to 5. Furthermore, the minimum rating value is 3, while the max is 5, and the ratings are given as decimal values. We can also see that the standard deviation is low at approximately 0.2, which means that majority of the data is clustered. Additionally, the mean is 4.8, further solidifying the fact that the data is left skewed and is not of uniform distribution.

These findings have led us to the conclusion that using the Kruskal-Wallis test is the best evaluation method for this hypothesis. This decision is supported by the following reasons:

  • Since we want to observe the effect of the ratings on the number of guests and the ratings are floating point values, we can bin the ratings into three groups, also known as categories.
  • Each group will be a range of values. This helps us to satisfy a primary requirement of the Kruskal Wallis test which is that there must be a single categorical independent variable, and a single continuous dependent variable. In this case, the rating is the independent variable and the number of guests is the dependent variable.
  • There are more than 2 rating groups, making this ideal from Kruskal Wallis.
  • The distribution is not normal.
  • The data is skewed.
  • There is low variance among the distribution of ratings.
  • The rating groups will be unequal in size, since most of the ratings fall between the 4.5 to 5 range.
  • Kruskal Wallis uses the median to compare groups, as opposed to means that are used by tests like the ANOVA. The mean of ratings in this distribution is impacted by the skew, making ANOVA an invalid option to use. Conversely, Kruskal Wallis is robust to skewed, non-uniform, low-variance, unevenly sampled, and overall unbalanced data. It is a non parametric test.

For these reasons, Kruskal Wallis is the best test to use for this hypothesis. Now, we can begin writing the code for this test.

res = []

# Break the ratings into 3 groups
for index, row in df.iterrows():
  if row['rating'] >= 3 and row['rating'] < 4.25:

Model 1: Can Rating Be Used to Predict Price?

NOTE: Partially inspired by ChatGPT when brainstorming the idea for this question

According to the data exploration, there exists a linear correlation between the numerical variables "Price" and "Rating." As a result, it may be possible to use a LinearRegression model to predict one feature based on the other. However, since a machine learning model was already used to predict many of the ratings used for this analysis, it would be inadvisable to once again predict ratings. Therefore, this analysis will use "Rating" to predict "Price."

It is important to note that "Price" is right-skewed (discovered during data exploration). Due to this, a log transformation was applied to the "Price" values in order to reduce the skew value and make them easier to use for modeling.

Training & Testing the Model

model_one_training.py 

  # Addressing price outliers with log transformations
  df['price'] = np.log(df['price'])

  # Confirming that the adjusted price still possesses a linear correlation with rating by

  print(f"MSE for Testing Data: {mse_test}")
  print(f"R2 Score for Training Data: {r2_train}")
  print(f"R2 Score for Testing Data: {r2_test}")

Analysis:
MSE: The training and testing MSE values are close to each other, which suggests this model is not severely overfitting; the model generalizes consistently -- but consistently poorly.

RMSE: On average, predictions are off by about 80-85 minutes. This is a large error relative to typical check-out times, indicating low accuracy.

MAE: Predictions are wrong by 41 minutes on average. Nearly identical train/test MAE shows the model is stable but weak.

R2: The resulting R2 score being close to 0 shows that the check-in and check-out times possess a weak correlation.

Insights & Conclusion

Key Takeaways

  • Model 1 Insight: Based on the MSE results, it appears that it is possible to use an Airbnb's rating to get a rough estimate of the location's price. However, by analyzing the graph and R-squared values, it is evident that most of the data being used for this analysis have very similar ratings (~4.8). As a result, despite the existence of a linear correlation between price and rating, the similarity between the data points used for this analysis is likely causing the model to depend less on the "Rating" variable to make a price prediction, and more so on memorizing the training data (which highly resembles the testing data numerically).

  • Model 1 Conclusion: Based on this analysis, an Airbnb's rating can be used to predict the Airbnb's price. However, these findings should be confirmed by repeating the analysis with a more diverse set of locations (especially in terms of ratings).

  • Model 2 Insight: By examining this plot, it becomes evident that using only a single feature provides limited predictive power. When relying solely on check-in time, the model struggles to accurately predict check-out time, as the relationship between the two variables appears weak and highly variable. The predicted values tend to cluster around the mean rather than follow the clear trend, indicating underfitting. This suggests that one feature alone is insufficient to capture the underlying patterns in the data. To better represent the complexity of check-out time behavior, additional features are needed to improve model performance and explanatory behavior.

  • Model 2 Conclusion: Although the training and testing errors are similar, indicating stable model performance, the model explains almost none of the variance in check-out time. This poor explanatory power is likely due to the use of only a single predictor, check-in minutes. Incorporating additional relevant features -- such as hotel location, city, or other operational factors -- would likely improve the model's ability to capture variability in check-out time.
